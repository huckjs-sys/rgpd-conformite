# RGPD Compliance — Roadmap

This document is the implementation plan for the `rgpd-conformite` plugin. It is the single source of truth for scope, ordering, and technical decisions. Update it as the project evolves.

---

## 1. Vision

Provide a community plugin that lets a ChurchCRM-based association meet the baseline obligations of the EU General Data Protection Regulation (GDPR / RGPD), without forking the core. Heavy inspiration is taken from [phili67/ecclesiacrm](https://github.com/phili67/ecclesiacrm), which integrates GDPR features in core; the goal here is to deliver the same value as a drop-in plugin.

## 2. Non-goals

The following are **explicitly out of scope** for this plugin:

- Modifying the core ChurchCRM person / family deletion workflow (would require upstream PR).
- Hiding deactivated persons from non-DPO users (no hook exists in core to filter person queries).
- Token-per-file or UUID-per-attachment storage (architectural change to ChurchCRM).
- Replacing or wrapping the existing MailChimp integration.

If any of these become essential, the right move is an upstream PR to ChurchCRM that adds a hook, **not** a patch in this plugin.

## 3. Inspiration: what EcclesiaCRM ships in core

| Feature | EcclesiaCRM file (reference only) | Adapted to plugin? |
|---|---|---|
| DPO settings (`bGDPR`, `sGdprDpoSigner`, `sGdprDpoSignerEmail`, `iGdprExpirationDate`) | core `SystemConfig` | ✅ via `plugin.json` settings |
| Audit trail per person | core hook into save/update/delete | ✅ via `PERSON_*` / `FAMILY_*` hooks |
| Data structure register (justify every field) | `src/EcclesiaCRM/model/EcclesiaCRM/GdprInfo.php`, `src/v2/templates/gdpr/gdprdatastructure.php` | ✅ plugin table + admin UI |
| Deactivated persons → DPO-only after N years | core query filters | ❌ requires core hook (out of scope v0.1) |
| MailChimp lists auto-flagged GDPR | core integration | ❌ out of scope |

## 4. Versioning plan

### v0.1.0 — MVP (target: working plugin scannable + installable)

Four self-contained features that deliver real GDPR value without core changes:

1. **DPO settings** — store DPO name, email, retention period in plugin config.
2. **Processing register** — admin UI listing every personal data field collected by ChurchCRM, each with an editable justification (Art. 30 GDPR).
3. **DSAR (right of access)** — given a Person ID, export every datum the CRM holds about that person as a structured JSON download (Art. 15 GDPR).
4. **Audit log** — listen for `PERSON_*` / `FAMILY_*` hooks, persist a tamper-evident log of who changed what when (Art. 5(2), Art. 30).

### v0.2.0 — Reach (target: post upstream feedback on v0.1)

- **Consent log** — explicit consent records per person (newsletter, photo, …) (Art. 7).
- **Retention policy** — scheduled task that flags / anonymises persons inactive for N years.
- **DSAR PDF export** — same data, printable format.
- **Data breach notification template** — pre-filled email template referencing Art. 33.

### v1.0.0 — Stabilisation

- Full FR / EN translations
- Cypress tests for the four MVP flows
- `SECURITY.md` audit completed
- Plugin scanner: zero warnings (justify or remove every `file_put_contents` etc.)

## 5. v0.1.0 detailed plan

### 5.1 DPO settings

**Goal:** centralise DPO contact details and retention defaults in plugin config so other features can read them.

**Implementation:**
- Add fields to `plugin.json` `settings`:
  - `dpoName` (text, required)
  - `dpoEmail` (text, required)
  - `retentionYears` (integer, default 2)
  - `enabled` (toggle, default false — kill switch)
- Read via `$this->getConfigValue('dpoName')` etc.
- Surface in the `/plugins/rgpd-conformite/settings` page (auto-generated by ChurchCRM from the manifest, but can be overridden with a custom view).
- The DPO email is appended to all DSAR exports for traceability.

**Files:**
- `plugin.json`
- `views/settings.php` (optional, for a richer form)
- `src/RgpdConformitePlugin.php`

**Acceptance:**
- Admin can fill DPO details, save, see them persisted across requests.
- Other features (DSAR, audit log) read these values correctly.

---

### 5.2 Processing register

**Goal:** comply with Art. 30 by maintaining a written register of processing activities. Admin enumerates every personal-data field ChurchCRM collects and writes a justification ("why we collect this, how long we keep it, who we share it with").

**Implementation:**
- New plugin-owned table:
  ```sql
  CREATE TABLE rgpd_processing_register (
      id INT PRIMARY KEY AUTO_INCREMENT,
      field_path VARCHAR(255) NOT NULL UNIQUE,   -- e.g. "person.email", "family.address"
      data_category VARCHAR(64) NOT NULL,         -- "identity", "contact", "financial", "sensitive"
      legal_basis VARCHAR(64) NOT NULL,           -- "consent", "contract", "legal-obligation", "legitimate-interest"
      retention_period VARCHAR(64) NOT NULL,      -- free text, e.g. "10 years for fiscal records"
      justification TEXT,                          -- free text from admin
      shared_with VARCHAR(255),                   -- e.g. "MailChimp, banque"
      updated_at DATETIME NOT NULL,
      updated_by INT NOT NULL                      -- person_id of the admin who last edited
  );
  ```
- Migration runs on plugin activation (via `boot()` checking schema version).
- Pre-seed with the standard ChurchCRM personal-data fields (introspected from `person_per`, `family_fam` Propel models).
- Admin UI (`/plugins/rgpd-conformite/register`):
  - DataTable listing every row, edit-in-place for `justification` / `legal_basis` / `retention_period`.
  - "Reset to default" button (re-runs the introspection).
- Export the register as PDF (Art. 30 deliverable for an audit).

**Files:**
- `src/RgpdConformitePlugin.php` (boot: schema migration)
- `src/Migrations/001_create_processing_register.php`
- `routes/routes.php` (`/register` GET + POST)
- `views/register.php`
- `src/Services/ProcessingRegisterService.php`

**Acceptance:**
- On first activation, the register is auto-populated from ChurchCRM's standard schema.
- Admin can edit, save, and the changes survive plugin disable/enable.
- A "Print register" action produces a clean PDF.

---

### 5.3 DSAR — Data Subject Access Request

**Goal:** given a Person ID, produce a complete JSON dump of every personal datum the CRM holds about that person, suitable for handing over to the data subject under Art. 15.

**Implementation:**
- Admin route `/plugins/rgpd-conformite/dsar/{personId}` (GET).
- Service layer aggregates from Propel models:
  - `Person` core fields
  - `Family` membership and family fields
  - `Address` (via family)
  - `Email` history
  - `GroupMember` (which groups they belong to)
  - `Note` (internal notes about them)
  - `EventAttendee` (attendance history)
  - `Donation` (financial donations — present, with disclaimer about retention obligations)
- Output a JSON object `{ subject: {...}, exportedAt: ISO8601, exportedBy: dpoName, byCategory: { ... } }`.
- Logged in audit log as `dsar.export`.

**Files:**
- `routes/routes.php` (`/dsar/{personId}` GET)
- `src/Services/DsarService.php`
- `views/dsar-form.php` (search-by-id form for the admin)

**Acceptance:**
- A test person with full data (memberships, donations, notes) returns every datum in one JSON, no PII missed.
- The export is downloadable as `dsar-{personId}-{date}.json`.
- The action is recorded in the audit log.

### 5.4 Audit log

**Goal:** record every personal-data lifecycle event so the DPO can demonstrate accountability (Art. 5(2)).

**Implementation:**
- New plugin-owned table:
  ```sql
  CREATE TABLE rgpd_audit_log (
      id BIGINT PRIMARY KEY AUTO_INCREMENT,
      occurred_at DATETIME NOT NULL,
      actor_id INT,                  -- ChurchCRM user who triggered the event
      actor_email VARCHAR(255),
      action VARCHAR(64) NOT NULL,    -- "person.created", "person.updated", "dsar.export", ...
      target_type VARCHAR(64),        -- "person" | "family"
      target_id INT,
      summary TEXT                    -- redacted human-readable summary, no PII inside
  );
  ```
- Hooks subscribed in `boot()`:
  - `Hooks::PERSON_CREATED`, `PERSON_UPDATED`, `PERSON_DELETED`
  - `Hooks::FAMILY_CREATED`, `FAMILY_UPDATED`, `FAMILY_DELETED`
- Admin UI (`/plugins/rgpd-conformite/audit`):
  - DataTable, filterable by date / actor / target / action.
  - CSV export for the DPO.
- Important: log lines do **not** contain PII (only `target_id` and `summary` like "Email field updated"). Logging PII would defeat the purpose.

**Files:**
- `src/Migrations/002_create_audit_log.php`
- `src/Services/AuditLogService.php`
- `src/Hooks/PersonHooksListener.php`
- `routes/routes.php` (`/audit` GET, `/audit/export.csv` GET)
- `views/audit.php`

**Acceptance:**
- Creating, updating, deleting a person in the CRM produces one log row each.
- Triggering a DSAR export produces one `dsar.export` row.
- Export CSV is well-formed and openable in a spreadsheet.
- No PII appears anywhere in the log table.

## 6. Capability inventory (for `approved-plugins.json`)

Based on the v0.1.0 plan, the plugin will need:

| Permission | Why |
|---|---|
| `network.inbound` | Slim routes under `/plugins/rgpd-conformite/...` |
| `db.read` | Reads `person_per`, `family_fam`, donations, etc. for DSAR + register seeding |
| `db.write` | Writes plugin-owned tables: `rgpd_processing_register`, `rgpd_audit_log` |
| `hooks.person` | Subscribes to `PERSON_*` for the audit log |
| `hooks.family` | Subscribes to `FAMILY_*` for the audit log |

**Not declared** (kept out for risk classification):
- `fs.write` — all file output goes through the HTTP response (downloads), no write outside plugin dir.
- `network.outbound` — no third-party calls.
- `email.send` — v0.1 doesn't notify; v0.2 may add this.
- `secrets.store` — DPO email is contact info, not a credential.

**Risk level:** `medium`. Combination of `hooks.person` + `db.write` reaches PII; `low` is reserved for plugins that don't touch people data.

## 8. Risks and open questions

| Risk | Mitigation |
|---|---|
| ChurchCRM Propel models change between versions, breaking DSAR introspection | Pin to ChurchCRM 7.x in `minimumCRMVersion`; integration test on every release |
| Audit log table grows unbounded | Document rotation policy; out-of-scope for v0.1 |
| `PERSON_UPDATED` hook fires too often (every save), bloating log | Coalesce same-actor same-target events within a 5-second window in v0.2 |
| Plugin maintainers cannot delete a person to fulfil Art. 17 | Document the limitation; v0.2 introduces a soft-anonymise action that overwrites PII fields with `[anonymised]` placeholders |
| Schema migration on plugin activation may fail silently | Wrap in a transaction; surface failure in the admin UI |
